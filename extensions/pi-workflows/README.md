# workflows

Durable workflow runtime for repo-local coding automation.

`workflows` runs workflow specs that coordinate agents, backlog state,
worktrees, review, and verified merges. Each run persists its state in
SQLite, so workflows can wait, retry, stop, and resume instead of being
limited to a single agent turn.

It also registers a `workflow_script` tool for one-off dynamic workflows:
the model can write a short JavaScript script that fans out work across
subagents with `agent()`, `parallel()`, `pipeline()`, `phase()`, and
`log()`. Use dynamic scripts for ad hoc audits and multi-perspective
analysis; use named specs for durable repo automation.

## Verification

From `extensions/workflows/`:

```bash
npm run typecheck
npm run test
npm run smoke
npm run release-check
```

These checks cover the focused TypeScript surface, SQLite state helpers,
lease/backlog regression tests, workflow wait/resume behavior, and a small
extension registration smoke test.

The packaged checks currently require `nix`.

## Tool

### `workflow_script`

Runs a short deterministic JavaScript workflow script generated by the
model. The first statement must be literal metadata:

```js
export const meta = {
  name: 'inspect_project',
  description: 'Inspect the repository from multiple angles',
}

phase('Scan')
const results = await parallel([
  () => agent('Map the repository structure.', { label: 'repo map' }),
  () => agent('Find risky or unusual implementation areas.', { label: 'risk scan' }),
])

phase('Synthesize')
const summary = await agent(
  'Summarize these findings:\n' + JSON.stringify(results),
  { label: 'final synthesis' },
)

return { results, summary }
```

Available globals:

- `agent(prompt, opts)` — run a subagent and return its final text
- `parallel(thunks)` — run `() => agent(...)` thunks concurrently
- `pipeline(items, ...stages)` — fan items through sequential stages
- `phase(title)` — mark progress groups
- `log(message)` — append a workflow log line
- `args` — optional JSON input passed to the tool
- `cwd` and `process.cwd()` — current repository/session directory
- `budget` — token budget tracker

Dynamic scripts cannot import modules, call `require()`, access direct
filesystem/network APIs, use `Date.now()`, `new Date()`, `Math.random()`,
reflection/constructor APIs, `eval()`, or `Function()`. Scripts are parsed
with Acorn, executed in a resource-limited worker, and terminated on abort
or wall-clock timeout. They must call and await `agent()` at least once and
return a bounded JSON-serializable result.

Dynamic workflows default to user agents. Selecting project or combined
agent scope requires interactive confirmation because `.pi/agents` is
repository-controlled.

Runtime limits are configurable through `concurrency`, `maxAgents`,
`tokenBudget`, and `timeoutMs`; hard caps also apply to script, argument,
prompt, log, phase, and result sizes. The worker and static policy reduce
capability exposure, but are not a security boundary for deliberately
hostile JavaScript.

## Commands

- `/workflow list` — list built-in workflows and recent runs
- `/workflow run <name> <args>` — run a workflow by name
- `/workflow show <run-id>` — inspect a stored workflow run
- `/workflow stop` — create `.pi/workflows/.grind-stop`
- `/grind <task>` — start a grinder run
- `/grind status` — show stop-marker state, backlog counts, latest run/round summary
- `/grind list` — list recent grinder runs
- `/grind show <run-id>` — render one grinder run in human-readable form
- `/grind clean <run-id>` — remove a grinder run plus known worktrees/branches
- `/grind checkout <run-id> <candidate-id>` — checkout a candidate branch if the repo is clean
- `/grind stop` — create `.pi/workflows/.grind-stop`
- `/fuzz <focus area>` — focused adversarial/fuzz run

## Built-in workflows

### `fuzz`

Runs a single agent with a focused adversarial prompt. Intended to generate:

- regression fixtures under `tests/adversarial/`
- backlog items under `backlog/`
- a log entry under `tests/adversarial/LOG.md`

Example:

```text
/fuzz string escapes
```

### `fix-one`

A small reference harness built on top of the new workflow API. It intentionally uses the reusable primitives instead of the larger `grind` engine:

- creates one backlog item
- claims it as `in-progress`
- creates one isolated worktree
- runs one implementation agent
- optionally runs one verify command
- runs one reviewer agent
- moves the backlog item to `done` or `tried`
- optionally auto-merges the resulting commit

Examples:

```text
/workflow run fix-one Add retry logic to the GitHub client
/workflow run fix-one {task:'Add retry logic', verify:'bun test'}
```

Options:

- `task` — required
- `verify` — optional command to run in the candidate worktree and again after merge
- `agent` — implementation agent, default `build`
- `reviewerAgent` — reviewer agent, default `review`
- `baseRef` — optional base branch/ref for the worktree
- `autoMerge` — default `true`
- `agentScope` — `user`, `project`, or `both`, default `both`

### `grind`

Creates multiple git worktrees and runs a round-based backlog grinder with
strategy-biased implementation agents.

Current grinder behavior:

- SQLite-backed workflow state under `.pi/workflows/state.db`
- SQLite-backed backlog states (`open`, `in-progress`, `done`, `tried`)
- orphan recovery from `in-progress` back to `open`
- rounds until `rounds` limit or `.pi/workflows/.grind-stop`
- triage agent chooses picks (fallback: first N open items)
- each picked backlog item is moved through `open -> in-progress -> done|tried`
- agents may create follow-up backlog items through the backlog API
- candidate auto-commit inside each worktree when possible
- reviewer agent runs on each candidate worktree and can block merges
- judge agent can reorder the merge queue
- one rotating specialist runs each round (scholar -> adversary -> coverage -> profiler -> refactor -> mutator)
- specialists write durable markdown artifacts under `.pi/workflows/artifacts/`
- meta agent audits each round, writes a markdown audit artifact, and summarizes next steps
- initial candidate scoring
- serialized merge queue back into the base repo via cherry-pick
- post-merge verify gate (when `verify` is set)

Example:

```text
/grind Add retry logic to the GitHub client
```

Or with options:

```text
/grind {task:'Add retry logic', candidates:3, verify:'bun test', keepWorktrees:true, dryLimit:2}
```

Options:

- `task` — required
- `candidates` — 1..6, default `3`
- `verify` — optional command run in each worktree after the agent finishes
- `agent` — implementation agent name, default `build`
- `triageAgent` — backlog triage agent name, default `plan`
- `reviewerAgent` — candidate review agent name, default `review`
- `judgeAgent` — merge-queue ordering agent name, default `plan`
- `metaAgent` — round audit/meta agent name, default `plan`
- `agentScope` — `user`, `project`, or `both`, default `both`
- `baseRef` — base branch/ref for worktree creation
- `keepWorktrees` — keep candidate worktrees after completion, default `true`
- `rounds` — optional hard cap on the number of rounds
- `dryLimit` — stop after this many consecutive empty rounds, default `2`
- `autoMerge` — enable serialized merge queue into the base repo, default `true`
- `autoReview` — run reviewer agent on each candidate before merge, default `true`
- `enableSpecialists` — run one rotating specialist per round, default `true`

## Storage

Workflow state is stored in:

```text
.pi/workflows/state.db
```

Artifacts and logs are stored on disk:

```text
.pi/workflows/runs/<run-id>/
.pi/workflows/logs/<run-id>.log
```

Specialist/meta artifacts live in:

```text
.pi/workflows/artifacts/literature/
.pi/workflows/artifacts/adversarial/
.pi/workflows/artifacts/coverage/
.pi/workflows/artifacts/perf/
.pi/workflows/artifacts/refactor/
.pi/workflows/artifacts/mutator/
.pi/workflows/artifacts/meta/
```

Grind worktrees are created as sibling directories next to the repo:

```text
../<repo-name>-workflows/<run-id>/
```

This avoids nesting linked git worktrees inside the main working tree.

## Custom specs

The extension now has a first step toward a reusable workflow API:

- `extensions/workflows/api.ts` defines a generic `WorkflowSpec` contract (`parseInput`, `createRun`, `step`)
- `extensions/workflows/registry.ts` provides a small registry for built-in and custom workflows

You can currently load custom workflows from `.pi/workflows/specs/` in two ways.

### 1. Simple JSON specs

For lightweight single-agent workflows, add JSON specs under:

```text
.pi/workflows/specs/*.json
```

Example:

```json
{
  "name": "research-api",
  "description": "Research a focused API topic",
  "mode": "agent",
  "agent": "explore",
  "prompt": "Investigate this topic and leave concise notes: {{args}}"
}
```

JSON specs are intentionally kept minimal and only support the simple `mode: "agent"` path. More advanced harnesses should use module workflows.

### 2. Module workflows (`.js`, `.mjs`, `.cjs`, `.ts`)

For custom harnesses built on the workflow API, add a module under:

```text
.pi/workflows/specs/*.{js,mjs,cjs,ts}
```

The module can export either:

- a default `WorkflowSpec`
- or `export const workflows = [ ... ]`

Example:

```ts
import type { WorkflowSpec } from "../extensions/workflows/api.js";

const spec: WorkflowSpec<{ task: string }, { phase: "run" | "done" }, { ok: boolean }> = {
  name: "research-one",
  description: "Minimal custom workflow module example",
  parseInput: (raw) => ({ task: raw.trim() }),
  createRun: async (input) => ({
    title: `research-one: ${input.task}`,
    summary: input.task,
    state: { phase: "run" },
  }),
  step: async (state, ctx) => {
    if (state.phase === "done") return { kind: "complete", state, result: { ok: true } };
    const result = await ctx.agents.run({
      agent: "explore",
      task: `Research this topic and summarize the key findings:\n\n${(await ctx.store.getRun()).argsText}`,
      cwd: ctx.cwd,
      agentScope: "both",
    });
    await ctx.artifacts.write("research-one", "notes.md", result.output);
    return {
      kind: result.ok ? "complete" : "failed",
      state: { phase: "done" },
      ...(result.ok ? { result: { ok: true }, summary: result.output.slice(0, 300) } : { error: result.stderr || "research failed" }),
    };
  },
};

export default spec;
```

Then run it with:

```text
/workflow run research-api HTTP caching semantics
/workflow run research-one HTTP caching semantics
```

If a module fails to load or exports an invalid workflow, `/workflow list` will show a module diagnostic entry so the failure is visible instead of being silently ignored.

## Notes

- `grind` uses git worktrees and workflow-run leases stored in `.pi/workflows/state.db`.
- `/workflow stop` and `/grind stop` write `.pi/workflows/.grind-stop` for long-running workflows to observe.
- `grind clean <run-id>` refuses to clean an active run.
- This is still an MVP: `fuzz` is single-agent, and `grind` now has backlog, specialists, review/judge, and merge-queue primitives, but it does **not** yet implement the full infinite swarm architecture or advanced scoring/perf heuristics.
